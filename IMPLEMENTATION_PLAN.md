# Complete implementation plan

Single document: product behavior, architecture, data model, APIs, frontend, AI, navigation/resume, phases, and delivery. Stack: **React (Vite, latest)**, **Python 3.x (latest)**, **PostgreSQL**, **OpenAI** (API key and model from config/env).

### v1 scope: one child per user and Start Over

- **Single profile** — Each logged-in user has **at most one child profile** in the database (e.g. name **“Rahul”**). There is **no** multi-child picker or list in v1.
- **Every login** — After authentication, the app loads **that one profile and its journey** only. `GET /journey/state` always reflects this single record (or an empty state if the user has never completed onboarding / has reset).
- **Start Over** — A **Start Over** control **deletes all persisted data** for that user’s journey: onboarding answers, module answers, image selections, `ai_runs`, `three_month_plans`, `module_sessions`, parent/journey inputs, and the **single `child_profile` row**. In one transaction, remove dependent rows (respect FK order or use `ON DELETE CASCADE`), then recreate a clean **`child_journey`** (and optionally a new empty **`child_profile`**) with `current_step_key` at the **beginning**. The user enters a **new child name** and builds a **new** profile from scratch.
- **Continue Onboarding** — When data exists, continues from **`current_step_key`** without wiping the database.

## Part A — Product scope (from requirements)

### A.1 Entry and onboarding

1. **Home** — User sees entry; actions: **Continue Onboarding** (resume using the **single** stored child + journey) or **Start Over** (full DB reset for that user as above, then start from the beginning).
2. **Intermediate screen** — Proceed to **Let’s Begin**.
3. **Chat / intake** — Conversational flow:
   - Questions **from backend** only; types: **free text** and **single choice** (buttons; one selection).
   - Each Q/A **persisted** as user progresses.
   - **Child’s name** (e.g. Rahul) captured and reused in copy on later screens.
   - **Reset** — Clears **this flow’s** answers in DB for current journey and restarts from first question (confirm UX).
   - **Mic** — Voice → text into input (browser Web Speech API or later server STT).
   - **Send** — Submits text answer.

### A.2 Personality analysis

4. After last onboarding answer — Message **“Let’s do a personality analysis…”** with loading animation (spec: ~2s first phase, then second loading while AI runs).
5. **OpenAI** evaluates all onboarding Q&A → **personality result** stored; screen shows analysis.
6. **Back** — Returns to **chat** with **all prior Q&A loaded from DB**; user sees **Let’s do a personality analysis** button (manual re-run, not auto). Behavior is **step-driven** (see Part E).

### A.3 Child profile (AI-generated)

7. **Continue** — AI builds **child profile** from analysis + Q&A; loading, then **profile detail** screen. Content **stored** after generation.
8. **Back** — Returns to **personality analysis** screen; data from DB.

### A.4 Growth areas hub

9. **Continue Now** — **Six** clickable sections (e.g. Critical Thinking, Creativity, …). User can enter each; progress **per module** persisted.

### A.5 Per module (repeatable pattern)

10. **~5 questions** — Mix of **options** and **text**; definitions from backend.
11. **See Summary** — Lists Q&A; **three** actions (per spec: e.g. explore activity, others).
12. **Explore Child Activity** — Primary CTA changes to **Present a fun game to {childName} on the same topic**.
13. **Image selection** — **Six** images; **exactly three** multi-select; **Submit My Choice** enabled only when 3 selected. Images have **text/caption** for AI.
14. Submit — **Generating Recommendations…** then OpenAI uses **child profile + module Q&A + three selected images** → **recommendations** screen; **persist** output.
15. Screen offers actions such as:
    - **Generate Recommendations** — Loading; AI produces **3-month** recommendations; **persist**.
    - **Explore More Growth Areas** — Return to **six sections**; user can pick another module and repeat.

### A.6 Life journey and dashboard

16. **Go to Life Journey** — Detail screen; **Back** per graph.
17. **Continue Journey** — Modal: parent name = **logged-in user**, child name = **profile**; parent **must** enter text; **Submit** → second modal → **Go to Dashboard**.
18. **Dashboard** — **3 months** in **3 sections**; each section **2 sub-sections**; each sub-section **2 weeks** of goals; **expand/collapse**. All from **stored** plan (generated by AI or seeded).

### A.7 Global rules (must hold everywhere)

- **Back** — Previous **logical** step; **every field and AI block** rehydrated from **PostgreSQL**, not from memory-only state.
- **Next login** — Resume at **`current_step`** for **the user’s only child profile / journey** with full state from DB.
- **AI** — Each generation **saved** (inputs + structured output); revisiting a screen **reads** saved data unless user explicitly **regenerates**.

## Part C — Step graph and journey state (single source of truth)

### C.1 `step_key` enumeration (example — adjust names to taste)

Define a **finite set** of steps; every screen maps 1:1 to one `step_key`.

**Onboarding / core**

- `home`
- `onboarding_entry` (after Continue or after **Start Over**)
- `onboarding_chat`
- `onboarding_personality_loading`
- `onboarding_personality_result`
- `child_profile_loading`
- `child_profile_result`
- `growth_hub`

**Growth module** (parameterize `module_key`: `critical_thinking`, `creativity`, …)

- `growth_{module}_quiz`
- `growth_{module}_summary`
- `growth_{module}_activity_cta`
- `growth_{module}_image_pick`
- `growth_{module}_reco_loading`
- `growth_{module}_reco_result`
- `growth_{module}_quarterly_loading`
- `growth_{module}_quarterly_result`

**Journey / dashboard**

- `life_journey`
- `journey_modal_1` / `journey_modal_2` (or single step with substates in JSON — prefer explicit steps if simpler)
- `dashboard`

Store **`current_step_key`** (+ optional `step_params` JSON, e.g. `{ "module": "critical_thinking" }`) on **`child_journey`**.

### C.2 Back edges

Maintain **`back_step_key`** per step in **config** (Python module or DB seed table `journey_step_def`: `step_key`, `back_step_key`, `flow_group`). **Back** = move to `back_step_key`, update `child_journey.current_step_key`, return full state payload.

Branching (e.g. from `growth_hub` into a module) sets `back` from module quiz → `growth_hub`.

### C.3 Resume

On login: load **the user’s single `child_journey`** (and **`child_profile`**, if present) → read **`current_step_key`** → **`GET /journey/state`** returns everything needed to render that step → frontend **syncs URL** to matching route. No `child_profile_id` query parameter in v1.

## Part D — PostgreSQL schema (structured)

### D.1 Identity

- **`users`** — `id`, `email`, `password_hash`, `display_name` (parent name), `created_at`, …
- **`child_profiles`** — `id`, `user_id`, `name` (e.g. **Rahul**), `created_at`, `updated_at`  
  - **v1: exactly one row per user** — enforce **`UNIQUE(user_id)`**. **Start Over** deletes this row (and all dependent data) or replaces it after a full wipe.

### D.2 Journey anchor

- **`child_journeys`** — `id`, `user_id`, `child_profile_id` (FK; nullable until the first saved name if you defer creating the profile row), `current_step_key`, `step_context` JSONB (e.g. `{ "module_key": "critical_thinking" }`), `status` (`active` / `completed`), `updated_at`  
  - **v1: exactly one journey per user** — enforce **`UNIQUE(user_id)`**. All `onboarding_answers`, `ai_runs`, `module_sessions`, etc. hang off this `journey_id`.

### D.3 Content definitions (versioned)

- **`flows`** — `id`, `key` (`onboarding`, `growth`, …), `version`
- **`questions`** — `id`, `flow_id`, `flow_version`, `key`, `ordinal`, `type` (`text` | `single_choice` | …), `prompt`, `metadata` JSONB  
- **`question_options`** — `id`, `question_id`, `label`, `value`, `ordinal`

### D.4 Answers

- **`onboarding_answers`** — `journey_id`, `question_id`, `text_value` nullable, `option_id` nullable, `updated_at`, **UNIQUE(journey_id, question_id)**

### D.5 AI runs (append-only; “latest” by `created_at` or `is_current` flag)

- **`ai_runs`** (generic) **or** separate tables per type:

  **Option A — one table**

  - `id`, `journey_id`, `run_type` (`personality` | `child_profile` | `module_reco` | `quarterly_plan` | …), `module_session_id` nullable, `model`, `input_snapshot` JSONB, `output_raw` TEXT, `output_json` JSONB, `status`, `error_message`, `created_at`

  **Option B — normalized** (clearer reporting): `personality_runs`, `profile_runs`, `recommendation_runs`, `quarterly_plan_runs` with FKs to `journey_id` / `module_session_id`.

Pick one; **implementation-ready** choice: **generic `ai_runs` + `run_type`** for speed, add indexes on `(journey_id, run_type, created_at DESC)`.

### D.6 Growth modules

- **`module_sessions`** — `id`, `journey_id`, `module_key`, `status`, `started_at`, `completed_at`  
  - **UNIQUE(journey_id, module_key)** if only one active attempt, or allow multiple with `attempt_no` if you need history.

- **`module_answers`** — `module_session_id`, `question_id`, `text_value`, `option_id`, **UNIQUE(module_session_id, question_id)**

- **`module_image_selections`** — `module_session_id`, `media_asset_id`, **UNIQUE**; enforce **exactly 3** in application layer or DB check constraint with trigger.

### D.7 Media

- **`media_assets`** — `id`, `module_key`, `image_url`, `caption_text`, `ordinal`, `metadata` JSONB

### D.8 Dashboard / plans

- **`three_month_plans`** — `id`, `journey_id`, `source_ai_run_id`, `plan_json` JSONB (strict schema: months → sections → subsections → weeks → goals), `created_at`  
  - Alternatively normalize: **`plan_months`**, **`plan_sections`**, **`plan_goals`** — use if you need SQL reporting on goals.

### D.9 Journey / parent input

- **`journey_events`** or **`parent_journey_inputs`** — `journey_id`, `step_key`, `input_text`, `created_at` (for modal text and audit)

### D.10 Optional

- **`journey_navigation_log`** — audit: `journey_id`, `from_step`, `to_step`, `reason` (`advance` | `back` | `resume`), `created_at`

### D.11 Indexes

- `(child_journeys.user_id, child_journeys.status)`  
- `(onboarding_answers.journey_id)`  
- `(module_sessions.journey_id, module_key)`  
- `(ai_runs.journey_id, run_type, created_at DESC)`

## Part E — Backend API (REST, implementation-ready)

**Auth**

- `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout` (as needed)
- All journey endpoints: **Bearer JWT**

**Bootstrap**

- `GET /me` — user + **the** single `child_profile` (if any) + journey summary

**Definitions**

- `GET /flows/onboarding/questions` — ordered questions + options  
- `GET /flows/growth/{module_key}/questions`  
- `GET /flows/growth/{module_key}/media` — six images

**Journey state (core)**

- `GET /journey/state` — resolves **the authenticated user’s** lone journey/profile (no child id query in v1). Returns:
  - `current_step_key`, `step_context`
  - `child_profile` (name, ids) or `null` after **Start Over** until a new name is saved
  - **Step payload** (discriminated union), e.g.:
    - `onboarding_chat`: list of `{ question, answer }` for replay
    - `personality_result`: `output_json` from latest successful `ai_runs` where `run_type=personality`
    - `growth_hub`: list of 6 modules + `module_sessions` completion flags
    - `image_pick`: assets + current selection ids
    - `dashboard`: `plan_json`

- `POST /journey/start` — body: `{ mode: "continue" }` — ensures `child_journey` exists; does **not** wipe data.

- `POST /journey/start-over` — **full reset** for the current user: in one transaction, delete all journey-related rows (`onboarding_answers`, `module_*`, `ai_runs`, `three_month_plans`, `parent_journey_inputs`, etc.), delete **`child_profile`**, then recreate an empty **`child_journey`** with `current_step_key` = `home` or `onboarding_entry`. Response matches initial `GET /journey/state`.

**Onboarding answers**

- `PUT /journey/onboarding/answers` — `{ question_id, text_value?, option_id? }` — upsert  
- `POST /journey/onboarding/reset` — delete onboarding answers for journey; set step to `onboarding_chat`; optional delete dependent `ai_runs` (policy: usually **soft-delete** or mark invalid; document choice)

**Advance / back**

- `POST /journey/advance` — body: `{ from_step, to_step? }` — server validates transition, updates `current_step_key`  
- `POST /journey/back` — server sets `current_step_key` to configured `back_step_key`, returns same shape as `GET /journey/state`

**AI triggers** (each writes `ai_runs` and advances step on success)

- `POST /journey/onboarding/personality/analyze`  
- `POST /journey/child-profile/generate`  
- `POST /journey/growth/{module_key}/recommendations` — body: selected `media_asset_ids` (length 3)  
- `POST /journey/growth/{module_key}/quarterly`  
- `POST /journey/plan/generate-dashboard` — if dashboard is separate from quarterly

**Module**

- `PUT /journey/growth/{module_key}/answers`  
- `PUT /journey/growth/{module_key}/images` — replace full set of 3 ids  
- `POST /journey/growth/{module_key}/reset` — optional, if module can be restarted

**Journey modals**

- `POST /journey/life-journey/continue` — body: `{ parent_note: string }` — saves input, advances step

**Errors**

- Consistent `{ code, message, details? }`; **409** if step transition invalid; **422** validation

**OpenAPI**

- Auto-generate from FastAPI; frontend generates types (openapi-typescript) optional.

## Part F — OpenAI integration

### F.1 Configuration

- Environment: `OPENAI_API_KEY`, `OPENAI_MODEL` (e.g. `gpt-4o`), optional `OPENAI_BASE_URL`.  
- **Never** log full prompts with PII in production without policy; log `run_id`, `journey_id`, token usage.

### F.2 Prompt strategy

- **System + developer** prompts versioned in code (`PROMPT_VERSION` in `ai_runs.input_snapshot`).  
- Request **JSON output** (JSON mode / schema) for: personality sections, profile fields, recommendations list, 3-month plan tree matching dashboard UI.

### F.3 Idempotency

- Client may retry; use **`Idempotency-Key`** header on POST AI endpoints mapping to dedup window or store last request hash on `ai_runs`.

### F.4 Failure

- On failure: `ai_runs.status=failed`, do not advance step; UI shows retry.  
- **Timeouts** (e.g. 60–120s) configurable.

## Part G — Frontend (Vite + React)

### G.1 Routes (map to `step_key`)

- `/`, `/onboarding`, `/chat`, `/personality`, `/profile`, `/growth`, `/growth/:module`, `/growth/:module/summary`, `/growth/:module/activity`, `/growth/:module/images`, `/growth/:module/recommendations`, `/growth/:module/quarterly`, `/life-journey`, `/dashboard`  
  Exact map is **derived from server `current_step_key`** on load.

### G.2 App shell

1. After login: **`GET /journey/state`**.  
2. If URL ≠ server step: **redirect** to canonical route for `current_step_key`.  
3. Provide **global Back** component calling **`POST /journey/back`** then replace cache from response.  
4. **Start Over** calls **`POST /journey/start-over`**, invalidates journey cache, redirects to home/onboarding.

### G.3 UI modules

- **Chat** — Renders from server list of Q&A; sends `PUT` answers; Reset → `POST .../reset`.  
- **Loaders** — First personality phase: **minimum display time** on client (e.g. 2s) **while** server runs AI (do not block server on fake sleep).  
- **Image grid** — Enforce 3 selections; `PUT` selection before submit; submit → `POST recommendations`.  
- **Dashboard** — Accordion from `plan_json`.  
- **Modals** — Controlled; on submit → `POST /journey/life-journey/continue` then refetch state.

### G.4 State

- **TanStack Query**: `journeyState` query key includes `journey_id` + `step_key`.  
- Invalidate after every mutation that changes step or answers.

### G.5 Mic

- `webkitSpeechRecognition` / `SpeechRecognition` with feature detection; graceful fallback “not supported”.

## Part I — Policies to document in README

- **Start Over**: **hard reset** for the user—remove the single `child_profile`, all answers, all `ai_runs`, plans, module progress, and reset `child_journey` to the initial step; document **FK delete / CASCADE** order in migrations.  
- **Single child (v1)**: **one** `child_profiles` row and **one** `child_journeys` row per `user_id`; no multi-child UX.  
- **Regenerate AI** (if added later): whether old `ai_runs` kept (recommended: keep for audit).  
- **Chat Reset** (in-flow): only clears onboarding answers for the current journey—**not** the same as **Start Over**.  
- **Mic**: browser support disclaimer.
